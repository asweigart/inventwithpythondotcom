<h3 class="h2" id="calibre_link-495"><span {http://www.idpf.org/2007/ops}type="pagebreak" id="calibre_link-326" class="calibre1"></span><strong class="calibre2">How to Play Hangman</strong></h3>
<p class="noindent">Hangman is a game for two people in which one player thinks of a word and then draws a blank line on the page for each letter in the word. The second player then tries to guess letters that might be in the word.</p>
<p class="indent">If the second player guesses the letter correctly, the first player writes the letter in the proper blank. But if the second player guesses incorrectly, the first player draws a single body part of a hanging man. The second player has to guess all the letters in the word before the hanging man is completely drawn to win the game.</p>

<h3 class="h2" id="calibre_link-498"><span {http://www.idpf.org/2007/ops}type="pagebreak" id="calibre_link-306" class="calibre1"></span><strong class="calibre2">Designing a Program with a Flowchart</strong></h3>
<p class="noindent">This game is a bit more complicated than the ones you’ve seen so far, so let’s take a moment to think about how it’s put together. First you’ll create a flowchart (like the one in <a href="#calibre_link-50" class="calibre4">Figure 5-1</a> on <a href="#calibre_link-51" class="calibre4">page 47</a> for the Dragon Realm game) to help visualize what this program will do.</p>
<p class="indent">As discussed in <a href="#calibre_link-52" class="calibre4">Chapter 5</a>, a flowchart is a diagram that shows a series of steps as boxes connected with arrows. Each box represents a step, and the arrows show the possible next steps. Put your finger on the START box of the flowchart and trace through the program by following the arrows to other boxes until you get to the END box. You can only move from one box to another in the direction of the arrow. You can never go backward unless there’s an arrow going back, like in the “Player already guessed this letter” box.</p>
<p class="indent"><a href="#calibre_link-53" class="calibre4">Figure 7-1</a> is a complete flowchart for Hangman.</p>
<div class="image"><img src="images/00081.jpeg" alt="image" class="calibre3" /></div>
<p class="figcap"><a id="calibre_link-53" class="calibre4"></a><em class="calibre6">Figure 7-1: The complete flowchart for the Hangman game</em></p>
<p class="indent"><span {http://www.idpf.org/2007/ops}type="pagebreak" id="calibre_link-872"></span>Of course, you don’t <em class="calibre6">have</em> to make a flowchart; you could just start writing code. But often once you start programming, you’ll think of things that must be added or changed. You may end up having to delete a lot of your code, which would be a waste of effort. To avoid this, it’s best to plan how the program will work before you start writing it.</p>
<h4 class="h3" id="calibre_link-499"><em class="calibre10"><strong class="calibre2">Creating the Flowchart</strong></em></h4>
<p class="noindenta">Your flowcharts don’t have to look like the one in <a href="#calibre_link-53" class="calibre4">Figure 7-1</a>. As long as <em class="calibre6">you</em> understand your flowchart, it will be helpful when you start coding. You can begin making a flowchart with just a START and an END box, as shown in <a href="#calibre_link-54" class="calibre4">Figure 7-2</a>.</p>
<p class="indent">Now think about what happens when you play Hangman. First, the computer thinks of a secret word. Then the player guesses letters. Add boxes for these events, as shown in <a href="#calibre_link-55" class="calibre4">Figure 7-3</a>. The new boxes in each flowchart have a dashed outline.</p>
<div class="image"><img src="images/00083.jpeg" alt="image" class="calibre3" /></div>
<p class="figcap"><a id="calibre_link-54" class="calibre4"></a><em class="calibre6">Figure 7-2: Begin your flowchart with a START and an END box.</em></p>
<div class="image"><img src="images/00084.jpeg" alt="image" class="calibre3" /></div>
<p class="figcap"><a id="calibre_link-55" class="calibre4"></a><em class="calibre6">Figure 7-3: Draw the first two steps of Hangman as boxes with descriptions.</em></p>
<p class="indent">But the game doesn’t end after the player guesses a letter. The program needs to check whether that letter is in the secret word.</p>
<h4 class="h3" id="calibre_link-500"><em class="calibre10"><strong class="calibre2">Branching from a Flowchart Box</strong></em></h4>
<p class="noindenta">There are two possibilities: the letter is either in the word or not. You’ll add two new boxes to the flowchart, one for each case. This creates a branch in the flowchart, as shown in <a href="#calibre_link-56" class="calibre4">Figure 7-4</a>.</p>
<div class="image"><span {http://www.idpf.org/2007/ops}type="pagebreak" id="calibre_link-873"></span><img src="images/00087.jpeg" alt="image" class="calibre3" /></div>
<p class="figcap"><a id="calibre_link-56" class="calibre4"></a><em class="calibre6">Figure 7-4: The branch has two arrows going to separate boxes.</em></p>
<p class="indent">If the letter is in the secret word, check whether the player has guessed all the letters and won the game. If the letter isn’t in the secret word, check whether the hanging man is complete and the player has lost. Add boxes for those cases too.</p>
<p class="indent">The flowchart now looks like <a href="#calibre_link-57" class="calibre4">Figure 7-5</a>.</p>
<div class="image"><img src="images/00089.jpeg" alt="image" class="calibre3" /></div>
<p class="figcap"><a id="calibre_link-57" class="calibre4"></a><em class="calibre6">Figure 7-5: After the branch, the steps continue on their separate paths.</em></p>
<p class="indent"><span {http://www.idpf.org/2007/ops}type="pagebreak" id="calibre_link-321"></span>You don’t need an arrow from the “Letter is in secret word” box to the “Player ran out of guesses and loses” box, because it’s impossible for the player to lose if they have just guessed correctly. It’s also impossible for the player to win if they have just guessed incorrectly, so you don’t need to draw an arrow for that either.</p>
<h4 class="h3" id="calibre_link-501"><em class="calibre10"><strong class="calibre2">Ending or Restarting the Game</strong></em></h4>
<p class="noindenta">Once the player has won or lost, ask them if they want to play again with a new secret word. If the player doesn’t want to play again, the program ends; otherwise, the program continues and thinks up a new secret word. This is shown in <a href="#calibre_link-58" class="calibre4">Figure 7-6</a>.</p>
<div class="image"><img src="images/00091.jpeg" alt="image" class="calibre3" /></div>
<p class="figcap"><a id="calibre_link-58" class="calibre4"></a><em class="calibre6">Figure 7-6: The flowchart branches after asking the player to play again.</em></p>
<h4 class="h3" id="calibre_link-502"><em class="calibre10"><strong class="calibre2">Guessing Again</strong></em></h4>
<p class="noindenta">The flowchart looks mostly complete now, but we’re still missing a few things. For one, the player doesn’t guess a letter just once; they keep guessing letters until they win or lose. Draw two new arrows, as shown in <a href="#calibre_link-59" class="calibre4">Figure 7-7</a>.</p>
<p class="indent">What if the player guesses the same letter again? Rather than counting this letter again, allow them to guess a different letter. This new box is shown in <a href="#calibre_link-60" class="calibre4">Figure 7-8</a>.</p>
<div class="image"><span {http://www.idpf.org/2007/ops}type="pagebreak" id="calibre_link-874"></span><img src="images/00092.jpeg" alt="image" class="calibre3" /></div>
<p class="figcap"><a id="calibre_link-59" class="calibre4"></a><em class="calibre6">Figure 7-7: The dashed arrows show the player can guess again.</em></p>
<div class="image"><img src="images/00094.jpeg" alt="image" class="calibre3" /></div>
<p class="figcap"><a id="calibre_link-60" class="calibre4"></a><em class="calibre6">Figure 7-8: Add a step in case the player guesses a letter they already guessed.</em></p>
<p class="indent"><span {http://www.idpf.org/2007/ops}type="pagebreak" id="calibre_link-323"></span>If the player guesses the same letter twice, the flowchart leads back to the “Ask player to guess a letter” box.</p>
<h4 class="h3" id="calibre_link-503"><em class="calibre10"><strong class="calibre2">Offering Feedback to the Player</strong></em></h4>
<p class="noindenta">The player needs to know how they’re doing in the game. The program should show them the hanging man drawing and the secret word (with blanks for the letters they haven’t guessed yet). These visuals will let them see how close they are to winning or losing the game.</p>
<p class="indent">This information is updated every time the player guesses a letter. Add a “Show drawing and blanks to player” box to the flowchart between the “Come up with a secret word” box and the “Ask player to guess a letter” box, as shown in <a href="#calibre_link-61" class="calibre4">Figure 7-9</a>.</p>
<div class="image"><img src="images/00095.jpeg" alt="image" class="calibre3" /></div>
<p class="figcap"><a id="calibre_link-61" class="calibre4"></a><em class="calibre6">Figure 7-9: Add a “Show drawing and blanks to player” box to give the player feedback.</em></p>
<p class="indent">That looks good! This flowchart completely maps out the order of everything that can happen in the Hangman game. When you design your own games, a flowchart can help you remember everything you need to code.</p>
<h3 class="h2" id="calibre_link-504"><span {http://www.idpf.org/2007/ops}type="pagebreak" id="calibre_link-305" class="calibre1"></span><strong class="calibre2">Summary</strong></h3>
<p class="noindent">It may seem like a lot of work to sketch out a flowchart about the program first. After all, people want to play games, not look at flowcharts! But it is much easier to make changes and identify problems by thinking about how the program works <em class="calibre6">before</em> writing the code for it.</p>
<p class="indent">If you jump in to write the code first, you may discover problems that require you to change the code you’ve already written, wasting time and effort. And every time you change your code, you risk creating new bugs by changing too little or too much. It is much more efficient to know what you want to build before you build it. Now that we have a flowchart, let’s create the Hangman program in <a href="#calibre_link-49" class="calibre4">Chapter 8</a>!</p>
